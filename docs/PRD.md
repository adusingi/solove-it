# PRD: 大切な人とやりたいことリスト

> **キャッチコピー：「忘れる二人を、お節介に繋ぎ止める。」**

---

## 1. 製品概要

### 1.1 プロダクト名

**「大切な人とやりたいことリスト」**

#### 名前候補

| 候補名 | 由来 |
| --- | --- |
| With You List（ウィズ・ユー・リスト） | 「あなたと一緒に」のリスト |
| アイリス（愛・List） | 愛 + List の掛け合わせ |
| Pair-Fill（ペアフィル） | Pair + Fill（満たす） |
| ふたリマインダー | ふたり + リマインダー |
| アスリスト | 明日 + Us + List |

### 1.2 プロダクトの説明

大切な人（恋人・パートナー・家族）と「いつかやりたいこと」を共有・管理し、**忘れがちな願いを"お節介な通知"で掘り起こす**スマートフォンアプリ。

既存の ToDo アプリが「タスクの消化」を促すのに対し、本アプリは **「感情の再燃」** を促す。

- ToDo リストのチェック → 「あぁ、終わった（解放）」
- **本アプリのチェック → 「あぁ、楽しかった（充足）」**

> **「やるべきこと」を管理するのではなく、「二人の温度感」を管理するツール。**

### 1.3 ビジョン

**「やりたいことを、やりたいときに、大切な人と。」**

世界中の「いつか行こうね」を一つでも多く叶える。

## 2. 課題と背景

### 2.1 解決する課題

「週末、何する？」「どこでもいいよ」「……（結局いつもの店）」  
— 誰もが経験のある **「不毛なやり取り」** 。

二人で話した「やりたいこと」は、**8 割が翌週には忘れられている**（体感値・仮説）。

### 2.2 心理学的根拠

#### ゼイガルニク効果（Zeigarnik Effect）の逆説

人間は「完了していないこと」を覚えている性質があるが、それは **「自分一人で完結できるタスク」** に限られる。  
二人で話した「やりたいこと」は、相手への期待値に依存するため、脳が **「今は保留（＝忘れていい情報）」** と判断しやすく、結果として記憶から抜け落ちてしまう。

#### 忘却曲線（エビングハウス）の適用

「あそこ行きたいね」という会話は、脳にとっては「単なるエピソード記憶」。  
学習（意味記憶）ではないため、**1 日後には 74% を忘れる** というデータがある。

### 2.3 既存ソリューションとの差別化

| 観点 | 既存 ToDo アプリ | 本アプリ |
| --- | --- | --- |
| 目的 | タスクの消化 | 感情の再燃 |
| チェック時の感情 | 「終わった（解放）」 | 「楽しかった（充足）」 |
| 管理対象 | やるべきこと | 二人の温度感 |
| 通知の性格 | リマインダー | お節介な Nudge |

## 3. ターゲットユーザー

### 3.1 メインペルソナ

- **恋人・夫婦**: 記念日や旅行のマンネリ解消。「いつか行こうね」を放置しがちなカップル。
- **親子**: 「いつか親を旅行に連れて行く」という、つい後回しにする願いを管理。

### 3.2 拡張ユースケース

- **家事分担への応用**: 「嫌な家事」をあえて Wish リストに入れて、通知を最大（鬼モード 👹）にしてゲーム感覚で消化。

---

## 4. コンセプトの 3 つの柱

### 柱 ① 非対称な期待の同期

「相手がやりたいと言ったこと」をメモするのは重いし、「自分の願望」を押し付けるのも気が引ける。

本アプリは **「いつか」という曖昧な状態を許容** し、**「二人の欲望の最大公約数」を勝手に見える化** する仕組みに価値がある。

→ **どちらかがリーダーシップを取らなくていい。**

### 柱 ② 埋もれた願いを掘り起こす（選べる通知：Nudge）

ユーザーが設定した「鬱陶しさレベル」に応じて、通知の頻度と文言を変化させる。

| レベル | ラベル | 頻度 | 通知例 |
| --- | --- | --- | --- |
| 0 😌 | 静か | 週1回 | 「今週の候補：{title}」 |
| 1 🙂 | 普通 | 週2回 | 「そろそろ『{title}』やらない？」 |
| 2 😤 | しつこい | 毎日 | 「まだ『{title}』が残ってるよ（圧）」 |
| 3 👹 | 鬼 | 1日2回 | 「『{title}』やるまで帰れま10🔥」 |

全然関係ない仕事中に、二人のスマホに同時に **「ねえ、焼肉忘れてない？」** と通知が届く。  
それをきっかけに「今日行く？」「行っちゃうか！」という **予定外の幸せ** が生まれる。

### 柱 ③ 達成を祝祭に変える（可視化）

達成した Wish を可視化し、二人の思い出として蓄積する。達成率ゲージやカテゴリ別の進捗表示により、「やりたいこと」の実現を祝祭的に演出する。

## 5. 機能要件

### 5.1 ペアリング

- 招待コードを送って二人を繋げる
- ペアリング時のワクワク感を演出する UI

### 5.2 Wish の登録・管理

- **カテゴリ選択**（10 カテゴリ）:
  - 🎉 記念日・イベント
  - 💕 デート
  - ✨ 体験・学び
  - 💪 健康・運動
  - 🧖 美容・リラックス
  - 🎨 趣味・創作
  - 🏠 家・暮らし
  - 💰 お金・ライフプラン
  - 👨‍👩‍👧 家族・友人
  - 🌍 社会貢献
- **優先度**: 高 / 中 / 低
- **時期**: 今月 / 来月 / いつか
- **予算レンジ**: 無料 / 〜5,000円 / 〜10,000円 / 〜30,000円 / 〜50,000円 / 50,000円〜
- **メモ**: 自由記述

### 5.3 フィルタリング・ソート

- 「今、予算 3,000 円以内でできることある？」といった検索が可能
- カテゴリ / 優先度 / ステータス でフィルタリング
- 優先度 / 時期 / 予算 / 作成日 でソート

### 5.4 通知設定（Nudge）

- 鬱陶しさレベル（0〜3）の設定
- 時間帯（朝 / 昼 / 夜）の選択
- 高優先度のみ通知するオプション
- 通知文言のランダム性によるエンターテインメント性

### 5.5 イベント連携

- 近隣のイベント情報を表示
- Wish に関連するイベントのレコメンド

### 5.6 達成管理

- Wish のステータス管理（未達成 / 達成済み）
- 達成率の可視化（プログレスバー）

## 6. 技術仕様

### 6.1 技術スタック

- **フレームワーク**: React Native (Expo)
- **言語**: TypeScript
- **ナビゲーション**: Expo Router（タブ + スタック構成）
- **状態管理**: React Context（WishProvider）

### 6.2 対応プラットフォーム

- iOS
- Android

### 6.3 技術的なこだわり

- **通知ロジック**: ユーザーが設定した「鬱陶しさ」をアルゴリズム化（頻度・文言のランダム性）
- **UI/UX**: 「ただのテキストリスト」ではなく、カテゴリ別アイコンや達成率ゲージでワクワクさせるデザイン

### 9.2 オープニング演出案

ペア（開発チーム）で登壇し、掛け合いから入る：

> **A**: 「ねえ B さん、来週何する？」  
> **B**: 「……忘れた。あ、これ（アプリ）見て！」

### 9.3 デモのハイライト

1. **ペアリング**: 招待コードを送って繋がる瞬間のワクワク感
2. **Wish の登録**: カテゴリ（旅行、食事など）を選んでサクッと登録
3. **フィルタリング**: 「今、予算 3,000 円以内でできることある？」と検索するシーン
4. **★ 通知設定（クライマックス）**: 実際にスマホで「鬱陶しい通知」が来る様子を実演
   - 全然関係ない仕事中に、二人のスマホに同時に「ねえ、焼肉忘れてない？」と通知が届く
   - それをきっかけに「今日行く？」「行っちゃうか！」という予定外の幸せが生まれる瞬間を見せる

### 9.4 プラスアルファのアイデア

#### ① ユーザーインタビュー風の演出

スライドに「実際に使ってみたカップルの声（あるいは自分たちの実体験）」を挟む。  
例：「通知がうざすぎて、逆に笑ってしまって、その勢いで旅行の予約をしました」

#### ② 「鬱陶しい通知」のバリエーション紹介

ハッカソンでは「遊び心」が評価される。通知文言のバリエーションを紹介して笑いを取る。

#### ③ デザインのビフォー・アフター

「ただのテキストリスト」だった初期案から、「カテゴリ別アイコンや達成率ゲージ」がついた現在の UI への進化を見せ、「ワクワクさせるための UI/UX の工夫」を語る。
